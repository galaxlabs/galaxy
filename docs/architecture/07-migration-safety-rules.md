# Migration Safety Rules

## Principles

1. **Metadata is source of truth.** The `tabDocType` + `tabDocField` rows define the target schema. Physical tables must match metadata.
2. **Never blindly alter.** Every migration step must be previewed before execution.
3. **Safe operations are automatic.** Dangerous operations require explicit confirmation.

## Safe Operations (Auto-Approve)

| Operation | DDL | Risk |
|-----------|-----|------|
| Create table | `CREATE TABLE IF NOT EXISTS "tabX" (...)` | None — idempotent |
| Add nullable column | `ALTER TABLE "tabX" ADD COLUMN IF NOT EXISTS col VARCHAR(255)` | None — existing rows get NULL |
| Add non-unique index | `CREATE INDEX IF NOT EXISTS ON "tabX" (col)` | Low — no data loss |

These are executed automatically during migration apply.

## Dangerous Operations (Require Preview + Confirmation)

| Operation | Risk | Reason |
|-----------|------|--------|
| Drop column | High | Irreversible data loss |
| Rename column | High | Breaks all existing queries |
| Change column type | High | May fail on incompatible data |
| Set NOT NULL | Medium | Fails if NULLs exist |
| Add UNIQUE | Medium | Fails if duplicates exist |
| Add foreign key | Medium | Locks table, fails if orphans exist |
| Delete table | Critical | Irreversible data loss |

These operations must be generated by the planner and shown to the user in preview format. The user must confirm before execution.

## Migration Flow

```
1. User modifies DocType metadata (adds/removes fields in Builder)
2. Save Metadata → writes to tabDocType + tabDocField
3. GET /api/migration/doctype/{name}/preview
   → plan_doctype_migration() reads metadata + current schema
   → Returns DDL plan showing what will be created/altered/dropped
4. POST /api/migration/doctype/{name}/apply
   → apply_doctype_migration() executes only safe operations
   → Dangerous operations are listed but NOT executed unless user explicitly approves
```

## Safety Checks in `apply_doctype_migration()`

The applier:
1. Compares metadata fields to existing `information_schema.columns`
2. Only runs `CREATE TABLE` for new tables
3. Only runs `ADD COLUMN` for new columns in existing tables
4. Does NOT drop columns, rename columns, or change types
5. Reports any divergent columns as warnings

## Core Table Safety

The 14 core tables in `core_tables.py` use `CREATE TABLE IF NOT EXISTS` only. They are never altered or dropped by the migration system. The `ALTER TABLE ADD COLUMN IF NOT EXISTS` pattern is used for backward-compatible schema additions (e.g., adding `tenant_id` to existing tables).

## Preview Output Format

```json
{
    "doctype": "Supplier",
    "table_name": "tabSupplier",
    "status": "preview",
    "plan": [
        "CREATE TABLE IF NOT EXISTS \"tabSupplier\" (name VARCHAR(255) PRIMARY KEY, supplier_name VARCHAR(255), ...)"
    ],
    "warnings": [],
    "columns_to_add": ["email"],
    "columns_to_drop": [],
    "columns_changed": []
}
```

## Frontend Safety Rule

**The frontend must never send raw SQL to execute.** All SQL plans are generated server-side by the migration planner. The frontend only sends:
- DocType name (for preview)
- DocType name + confirmation (for apply)

## Rollback

Current implementation has no automatic rollback. If a migration fails:
- `CREATE TABLE` is atomic per table
- Partial migrations may leave some tables created and others not
- Manual intervention required via SQL

Future: Wrap migration in a transaction and roll back on any failure.
